软硬协同优化--code-size优化

发表于 2025-08-06 分类于 优化 阅读次数： Disqus：

code-size：代码体积
代码体积的缩小有利于降低运行设备的功耗，提高程序性能【代码体积小，所占用的内存、寄存器等设备就少，从而达到降低功耗、提高性能的目的】。

常用的方式就是通过编译时加入-Os选项来优化。包括死代码消除、死参数消除、稀疏条件常量传播等。

在指令集层面，为了达到优化代码体积的目的，目前有许多方法：变长指令集、压缩指令、位域掩码立即数编码、寄存器编码重叠、融合指令、向量指令、硬件循环等方式。

  1. 变长指令集：对于使用频率高的指令，采用更短的编码，使用频率更低的指令，采用更长的编码，这样就能以尽可能小的体积容纳尽可能多的操作，从而达到压缩代码体积的目的。
  2. 压缩指令：在满足一定的特殊条件之后【例如指令本省表达的信息不多，16bit指令也能够表达等】，能够在编译阶段将32bit的指令压缩至16bit的指令，在硬件执行时再通过指令的标识位判断是否解码后再执行。一次达到压缩代码体积，减少硬件成本，提高执行效率的目的。
  3. 位域掩码立即数编码：使用13位编码表示一个有明显重复规律的数，例如0x00FF00FF00FF00FF，可以表示为00FF重复4次，这些信息就以某种规则存放在13位编码中，即可以实现立即数表示的代码体积压缩。但是位域掩码立即数编码不能表示所有64bit的立即数，需要立即数有明显重复规律。
  4. 寄存器编码重叠：将多个使用率较低的寄存器使用同一编码。但是，正由于多个寄存器使用同一编码，这些寄存器不能在一条指令中同时使用。可以通过特定指令➕该编码的形式区分寄存器。【例如0b1111既表示寄存器15，也表示寄存器16。但是只有在add指令下才是调用寄存器15，在mul指令下才是调用寄存器16。】其实也是将指令和寄存器编号合起来指令某一个寄存器，也能使用将使用频率低的寄存器用更复杂的编码方式表示这一原理解释。
  5. 融合指令：使用一条指令表达多条指令的意义。即将执行频率较高的指令组合，打包并给一个短编码。例如FMA指令，现将两操作数相乘再与另一个操作数相加；ABS指令直接输出绝对值，而非常规的通过硬件异或计算出绝对值。还有其他的一次性读写两个寄存器的load-store pair等指令。但是在硬件层面会增加流水线的复杂程度，而且，在编译器层面，要识别这些指令并选择对应的融合指令替代，也增加了编译器的复杂程度。其次，增加融合指令，就需要额外的编码表示这些指令，会压缩可扩展的空间。
  6. 向量指令：例如SIMD单指令多数据模式，可以用一条指令进行多条数据的处理，能够有效减少指令数量，从而减少代码体积。
  7. 硬件循环：将需要多次执行的一个或一组指令完全交由硬件执行。常规循环场景为硬件执行完后，更新循环变量，然后使用跳转指令跳转到循环体指令开始位置重新执行。当进行循环展开优化后，虽然不用或减少了更新和跳转指令数量，但是会占用大量静态code-size，code-size体积增大。硬件循环是告诉硬件需要执行多少次后，完全由硬件自主完成循环的过程。这样既不会增加静态代码的体积，也能保证循环变量和指令地址跳转的零开销。【虽然硬件内部也需要维护一个循环变量，也需要地址跳转，但是维护的循环变量是内部专用的寄存器维护，可以并行处理且不需要指令驱动，同理，地址跳转也是内部地址跳转，无指令开销，无寄存器开销】
    code-size的各种优化方法，虽然将代码体积降下去了，但是不可避免的增加了一些指令的复杂程度，例如融合指令，寄存器编码重叠等。变长指令集也是一个复杂编码的典型例子，指令是变长的意味着解码需要更加复杂的规则识别指令。
